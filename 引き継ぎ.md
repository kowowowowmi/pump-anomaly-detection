## Project 2の選定経緯（なぜこのプロジェクトか）

### 検討した選択肢と最終決定までの流れ

**最初の候補：NDFCブイ波高予測**
- 却下理由：波高は「USVが動かす前に確認する外部条件」であり、USVが実際に収集・処理するデータではない。MizLinxの本業（MizLinx Monitor：水質センサー異常検知 / AUV：水中画像AI解析）とは一段ずれている。さらにラベルなしデータでは異常検知の評価が主観的になる。

**途中で検討：多変量海洋センサー（NDBC/EPA水質データ）異常検知**
- 保留理由：ドメインの一致度は高いが、ラベルなしのためモデルの評価が「視覚的な確認」や「ドメイン知識による閾値設定」に頼ることになり、ポートフォリオとして数値的な説得力を出しにくい。

**最終採用：Pump Sensor Data（Kaggle）による異常検知**
- 採用理由：
  - 機械状態のラベル（NORMAL / BROKEN / RECOVERING）がついており、Precision・Recall・F1スコアで客観的にモデルを評価できる
  - 多変量（51センサー）の時系列データで、異常検知の手法（ARIMA残差・Isolation Forest等）を一通り学べる
  - 「工業用ポンプの異常」はUSVの推進システム・エンジンの予知保全と本質的に同じ問題であり、MizLinxへの応募時に「海洋ロボティクスの文脈に置き換えて語れる」
  - ラベルありのため、複数手法を比較して勝敗を明確に示せる

## プロジェクト概要

| 項目 | 内容 |
|---|---|
| データソース | Kaggle「Pump Sensor Data」（ブラウザからDL済み or DL予定） |
| データ規模 | 約22万行、52列（timestamp + sensor_00〜sensor_50 + machine_status） |
| ラベル | `NORMAL` / `BROKEN` / `RECOVERING` の3状態 |
| 予測タスク | 多変量センサーの時系列から機械の異常状態（BROKEN）を検知する |
| ポートフォリオ文脈 | 「USV・AUVの推進系予知保全と同じ問題設定を、ラベルありデータで定量的に解いた」ことをMizLinxへの応募で示す |
